# Spec: lazy-name-save

**Timestamp:** 20260627T140520Z  
**Pillar:** Signup / auth (Pillar 2)  
**Friction removed:** Magic-link users have no `full_name` in Supabase `user_metadata`; the
contact-prefill feature (`contact-prefill` cycle) therefore leaves the name field blank for
them on every post visit. They must type their name each time. This cycle silently saves the
`contact_name` they submit on their first partnership or seeking post to `user_metadata.full_name`
‚Äî on the next form visit the name is already there.

---

## Goal

Magic-link users who post a listing stop having to retype their name on every subsequent
visit to a post form. The first time they type and submit a `contact_name`, it's persisted
to their Supabase user metadata so the existing contact-prefill logic fills it in on future
visits ‚Äî with zero new UI and zero schema change.

---

## Scope

Files expected to touch:
- `src/app/actions.ts` ‚Äî in `createPartnership` and `createSeekerListing`, after successful
  insert, call `supabase.auth.updateUser({data:{full_name: contactName}})` when the user
  has no existing `full_name` and a `contact_name` was submitted.
- `src/components/PostPartnershipForm.tsx` ‚Äî add a one-line hint under the name field
  when `isLoggedIn && !userName` so the user understands the name will be remembered.
- `src/components/PostSeekerListingForm.tsx` ‚Äî same hint.

`createAircraftListing` is out of scope: the aircraft form uses `poster_id` + in-platform
messaging (no `contact_name` field), so there is no name to save from that flow.

---

## Acceptance criteria

1. After a magic-link user submits a partnership or seeking listing with a non-empty
   `contact_name`, `user_metadata.full_name` is set to that name.
2. On the next visit to `/partnerships/new` or `/partnerships/seeking/new`, the
   name field is pre-filled (via the existing `userName` prop + `defaultValue`).
3. Users who already have `user_metadata.full_name` set (Google OAuth users) are
   unaffected ‚Äî the save is conditional on `!user.user_metadata?.full_name`.
4. The hint text ("We'll save your name for future listings.") is visible on the
   name field only when the user is logged in but has no display name set yet.
5. `npx next build` passes with zero TypeScript errors.
6. QA smoke exits 0 on `/partnerships/new` and `/partnerships/seeking/new` at
   desktop 1280 + mobile 375 (HTTP 200, no app-origin console errors, no overflow).

---

## Out of scope

- Google OAuth users (they already have `full_name`).
- The aircraft post form (`/aircraft/new`) ‚Äî no contact name field; uses `poster_id`.
- Any changes to auth files (FREEZE.md).
- Schema changes.
- Visible success/confirmation UI after the save ‚Äî it's intentionally silent.
